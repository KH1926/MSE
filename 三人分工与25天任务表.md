# 三人分工与 25 天任务表

## 1. 分工原则

三人团队按“后端数据流 + 算法仿真流 + 前端文档流”分工，不建议平均分摊。

主线始终围绕：

`订单录入 -> 工单生成 -> 排产 -> 执行仿真 -> 质检分流 -> 成品入库 -> KPI分析`

## 2. 角色分工

### A：后端 / 数据库 / 主流程负责人

负责内容：

- `SQLite` 表结构
- 主数据维护
- 工位随机分布参数与工位数据落库
- 订单、工单、排产结果落库
- 事件记录与接口开发
- 虚拟产线事件数据格式定义
- 后端联调

核心交付：

- 数据库表
- `FastAPI` 接口
- 测试数据
- 主流程稳定性

### B：排产 / 仿真 / 实验负责人

负责内容：

- 排产规则
- 工位随机分布参数确定与瓶颈识别
- 工位节拍
- 离散事件仿真（使用 `SimPy`，不自定义 DES 框架）
- 真实约束参数抽象
- 质检返修逻辑
- `KPI` 统计
- 生产线平衡率分析
- 三组实验数据
- `D3-D4` 阶段额外输出：`/mock` 目录下的静态 `JSON` 文件（订单列表、工位状态、`KPI` 数据），供 C 在接口就绪前独立推进前端

核心交付：

- 排产逻辑
- `SimPy` 仿真逻辑
- 质检与返修逻辑
- 对比图表与实验结论
- `/mock` 静态 `JSON` 数据文件（`D4` 前完成）

### C：前端 / 看板 / 文档统稿负责人

负责内容：

- 前端模板筛选与改造（`D3` 直接按能力选定一个模板跑通首页，不需同时试跑两套）
- `5` 个核心页面
- 截图、图表、流程图、鱼骨图、节拍平衡图
- 数字孪生可视化表述与文献综述分层
- 文档统稿
- `D15` 完整系统跑通后立即编写演示脚本（固定演示路线、触发场景、截图节点）
- 答辩材料整理

核心交付：

- 页面展示
- 演示素材
- `D15` 完成的演示脚本（演示路线、截图节点、答辩口径）
- 论文/说明文档
- `PPT` 大纲

## 3. 协作边界

> 详细字段定义、状态枚举、接口规范、KPI 口径、术语对照，见 **团队协作契约.md**。本节只说明谁负责提供什么。

### A 提供给 B

- 订单数据
- 工单数据
- 工位与工艺路线数据
- 排产输入接口

### B 提供给 A

- 排产结果
- 仿真事件
- 质检记录
- `KPI` 统计结果

### A 与 B 提供给 C

- 页面要展示的字段
- 图表要展示的数据
- 可截图的稳定场景

### C 反向约束 A/B

- 页面字段命名
- 文档术语统一
- 截图和答辩所需数据格式

## 4. 检查点

- `D5`：数据库与流程图完成；`B` 与 `A` 确认 `work_order`、`schedule_task` 字段格式冻结（后续只新增字段，不改已有命名）
- `D7`：工位随机分布参数表与首版平衡图完成
- `D10`：订单到排产链路跑通
- `D12`：虚拟产线事件数据格式固定

## 7. 每日同步规则

每天晚上只同步 `3` 件事：

1. 今天主链路多跑通了什么
2. 明天哪个点最可能卡住
3. 文档是否同步更新

同步会议控制在 `15` 分钟内。

## 8. 分工红线

- A 不负责重前端开发
- B 不负责页面样式和排版
- C 不承担核心算法和数据库设计
- 任一模块若不能服务主流程，直接砍掉

## 9. 一句话总结

三人团队最稳的打法是：**A 保数据与流程，B 保算法与实验（`SimPy` + `D4` 前交付 mock 数据），C 保展示与文档（`D15` 锁定演示脚本）；25 天只围绕一条演示闭环推进，不做额外发散。**
