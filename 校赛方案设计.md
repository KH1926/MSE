# 本科生第四主题校赛方案设计

## 1. 项目定位

- 方向：本科生组-智能生产管控
- 形式：纯软件方案，不接入真实 `PLC`、`HMI`、机器人或实物产线
- 目标：围绕“智能小车”虚拟装配产线，构建一个可演示、可分析、可写稿的轻量闭环系统

## 2. 核心思路

以虚拟产品“智能小车”为对象，完成如下闭环：

`订单录入 -> BOM/工艺 -> 工单生成 -> 排产 -> 执行仿真 -> 质检分流 -> 成品入库 -> KPI分析`

方案重点只保留三部分：

- 轻量 `MES` 管控流程
- 排产与调度逻辑
- 仿真与优化分析结果

同时吸收精益生产中成本较低、收益较高的方法作为论文论证层：

- 工位随机分布参数表
- 节拍平衡图与瓶颈识别
- `人-机-料-法-环` 鱼骨图问题分析

结合论文检索工作流，进一步采用“真实约束仿真环境 + 动态调度场景 + 虚拟产线数据采集 + 数字孪生可视化”的表达方式，提高方案专业性。

说明：

- 上述方法主要用于分析与写作，不额外扩大系统开发范围
- `7S`、供应链优化、设备全生命周期管理等内容可作为文献背景，不纳入本次校赛核心实现
- `LLM`、强化学习、多智能体、神经组合优化等前沿方法仅作为研究背景和展望，不纳入当前实现

## 3. 虚拟产品与工位设计

### 3.1 产品定义

- 产品名称：智能小车
- `BOM`：底盘、轮子、芯片、外壳、螺丝包

### 3.2 工位设计

| 工位 | 名称 | 主要动作 | 作用 |
| --- | --- | --- | --- |
| `WS10` | 底盘上线 | 底盘进入产线 | 展示起始物流 |
| `WS20` | 核心组装 | 安装轮子与芯片 | 设为瓶颈工位 |
| `WS30` | 自动锁螺丝 | 完成装配 | 展示装配过程 |
| `WS40` | 视觉检测 | 合格/不合格判定 | 展示质检与分流 |
| `WS50` | 成品包装 | 合格品入库 | 展示终点物流 |

### 3.3 返修逻辑

- `WS40` 检测不合格后回流至 `WS30`
- 最多返修 `1` 次
- 二次不合格判定为报废

## 4. 系统模块

系统只保留 `6` 个核心模块：

1. 基础数据模块：产品、`BOM`、工位、工艺路线
2. 订单与工单模块：订单录入、工单生成
3. 排产调度模块：根据交期、优先级、瓶颈工位进行排产
4. 执行仿真模块：模拟工位加工、排队、完成、流转
5. 质量管控模块：质检、返修、报废记录
6. 看板分析模块：`KPI`、甘特图、进度监控、优化对比

补充说明：

- 不单独开发“精益管理平台”
- 精益分析内容通过工时表、平衡图、鱼骨图、优化前后对比表体现在文档和看板中

## 5. 专业性增强但不增加工作量的提取项

从论文研究工作流中提取以下低成本内容，用于提升文档和答辩专业性：

1. 真实约束建模：不做复杂强化学习环境，只在仿真参数中体现交期、瓶颈工位、插单、故障、返修等约束
2. 动态调度表达：将插单与故障场景表述为”动态调度与扰动响应”，实际实现仍采用规则重排产
3. 虚拟产线数据采集：把系统运行中的订单、工单、事件、质检、`KPI` 记录称为”虚拟产线事件数据”
4. 数字孪生可视化：将 `WS10-WS50` 工位状态、甘特图、看板称为”轻量数字孪生可视化”
5. 文献综述分层：背景部分按”生产调度、工厂仿真、数据采集、数字孪生可视化”组织，而不是堆论文
6. 演示脚本提前固定：系统仿真跑通后（`D15`）立即确定演示主路线（从哪条订单进、触发哪个场景、截哪张图），后续文档、截图、实验数据全部围绕该脚本对齐，避免封版时页面与文档描述不一致

## 6. 国赛软硬件在校赛中的体现方式

本方案不直接使用国赛实物平台，而是进行软件化映射：

- `MES软件`：由自研轻量 `MES` 页面体现
- `SQLite`：用于订单、工单、排产、事件、质检、`KPI` 数据存储（单文件，无需服务器，便于演示携带）
- `Python`：用于排产、仿真（`SimPy`）、统计分析和后端服务
- `PLC/HMI`：抽象为工位状态机和监控页面
- 机器人/自动工站：抽象为 `WS20`、`WS30` 的自动执行逻辑
- 视觉检测：抽象为 `WS40` 的质检判定与返修分流
- `Plant Simulation`：抽象为虚拟产线仿真与性能分析（由 `SimPy` 实现）

## 7. 技术路线

### 7.1 推荐技术栈

- 后端：`Python + FastAPI`
- 数据库：`SQLite`（纯演示系统，无需独立数据库服务器，携带单文件即可演示）
- 仿真：`SimPy`（工位资源模型与 `Resource` / 事件队列天然匹配，不使用自定义离散事件逻辑）
- 前端：优先复用开源模板，不从零开发

### 7.2 前端实现原则

前端是本项目最容易失控的效率陷阱，必须坚持以下原则：

- 绝对不要从零手写前端后台系统
- 直接复用现成的 `Vue3` 后台管理或大屏模板
- 只修改字段、表格、图表绑定和少量样式
- 把时间优先投入在排产、仿真、实验和文档上

### 7.3 实现方式

**模板化前后端分离**

- 前端：开源 `Vue3 Dashboard Template + ECharts`
- 后端：`FastAPI`
- 适用情况：队内有人会基本前端，能完成接口对接与页面改造

## 8. 页面设计

建议只做 `5` 个页面，且全部基于现成模板改造：

1. 首页驾驶舱：完工量、准时率、利用率、在制品、不良率
2. 订单管理页：订单录入、订单状态、工单生成
3. 排产中心页：排产结果、任务列表、甘特图
4. 生产监控页：工位状态、工单进度、事件日志
5. 质量分析页：合格率、返修率、异常统计、优化前后对比

页面目标是“展示结果”，不是自研完整前端系统。

## 9. 核心实验设计

只做 `3` 组场景：

1. 正常生产：验证系统主流程可跑通
2. 插单场景：验证动态调度与扰动响应能力
3. 故障场景：验证真实约束下的影响分析能力

每组实验建议固定输出：

- 工位随机分布参数表
- 节拍平衡图
- 瓶颈工位变化
- 关键 `KPI` 对比
- 虚拟产线事件数据摘要

## 10. 核心指标

论文和答辩重点围绕以下指标展开：

- 总完工量
- 订单准时率
- 平均生产周期
- 设备利用率
- 在制品数量
- 不良率 / 返修率
- 生产线平衡率
- 瓶颈工位作业时间

## 11. 开发顺序

建议按以下顺序推进：

1. 固定产品、工位、流程和指标
2. 建立工位随机分布参数表并做首版节拍平衡分析
3. 建数据库与主数据
4. 完成订单录入与工单生成
5. 完成基础排产逻辑
6. 完成执行仿真与质检分流
7. 固定虚拟产线事件数据格式
8. 选定并跑通现成前端模板
9. 只做必要页面改造与接口对接
10. 跑实验、出图表、写文档

开发原则：

- 算法优先于页面美化
- 模板复用优先于从零开发
- 演示闭环优先于功能堆砌

## 12. 最小可交付成果

校赛阶段至少完成以下内容：

- 一个可演示的纯软件系统
- 一条完整业务主线演示
- 三组实验结果与图表
- 一篇完整说明文档
- 一套工位随机分布参数表、节拍平衡图和瓶颈分析结果
- 一份虚拟产线事件数据说明

## 13. 一句话总结

本方案以“智能小车虚拟装配线”为对象，通过轻量 `MES + 排产 + 仿真 + 看板` 的方式，在纯软件条件下完成对智能生产管控能力的展示与验证；表达上采用“真实约束仿真、动态调度、虚拟产线数据采集、数字孪生可视化”的专业框架，实现上坚持“模板优先、算法优先、前端从简”，以适应当前校赛时间与资源约束。
